# TG Poster

Telegram-бот для автоматической публикации файлов (фото, видео, документы) в каналы по расписанию с веб-интерфейсом управления.

## 📋 Возможности

- **Веб-интерфейс** — управление очередями, каналами и мониторинг публикаций
- **Планировщик** — автоматическая публикация по времени (с учётом интервалов и jitter)
- **Гибкость** — настройка времени начала/окончания, интервала между постами
- **Мониторинг** — отслеживание статуса отправок, логов и ETA завершения
- **Тестирование** — отправка тестовых сообщений в каналы
- **Docker** — готовность к развёртыванию через Docker Compose

## 🚀 Быстрый старт

### Через Docker Compose (рекомендуется)

1. **Склонируйте репозиторий:**
   ```bash
   git clone <repository-url>
   cd tg_manager/tg_poster
   ```

2. **Настройте переменные окружения:**
   
   Отредактируйте `docker-compose.yml`:
   - Замените `BOT_TOKEN` на токен вашего бота
   - При необходимости измените пути к медиафайлам (`C:/tg_media`)
   - Измените порт маппинг если нужно (`7777:7777`)

3. **Запустите контейнер:**
   ```bash
   docker-compose up -d
   ```

4. **Откройте веб-интерфейс:**
   ```
   http://localhost:7777
   ```

### Локальный запуск (без Docker)

1. **Установите зависимости:**
   ```bash
   pip install -r requirements.txt
   ```

2. **Настройте переменные окружения:**
   ```bash
   export BOT_TOKEN="your-bot-token"
   export TZ="Asia/Yekaterinburg"
   ```

3. **Запустите приложение:**
   ```bash
   python -m app.main
   ```

4. **Откройте в браузере:**
   ```
   http://127.0.0.1:7777
   ```

## 📖 Использование

### 1. Добавление канала

1. Перейдите на главную страницу (`/`)
2. Введите `chat_id` канала (можно получить через [@userinfobot](https://t.me/userinfobot))
3. Укажите название канала
4. Нажмите «Добавить»

> ⚠️ Бот должен быть администратором канала с правом публикации сообщений.

### 2. Создание очереди

1. Перейдите на страницу очередей (`/queues`)
2. Выберите канал из списка
3. Настройте параметры:
   - **Название** — имя очереди
   - **Папка источника** — путь к файлам для публикации
   - **Время начала/окончания** — диапазон публикации (формат `HH:MM`)
   - **Интервал** — минимальное время между постами (сек)
   - **Jitter** — случайная задержка (сек) для естественности
4. Нажмите «Создать очередь»

### 3. Управление очередью

На странице управления очередью (`/queue/<id>/manage`):

- **Старт/Пауза** — запуск или остановка публикации
- **Принудительный старт** — немедленная активация
- **Тестовая отправка** — проверка связи с каналом
- **Логи** — история отправок (последние 50)
- **Удаление** — удаление очереди и её логов

## 🔧 Конфигурация

### Переменные окружения

| Переменная | Описание | Пример |
|------------|----------|--------|
| `BOT_TOKEN` | Токен Telegram-бота | `123456:ABC-DEF1234ghIkl-zyx57W2v1u123ew11` |
| `TZ` | Часовой пояс | `Asia/Yekaterinburg` |

### Параметры очереди

- **interval_sec** — базовый интервал между публикациями (мин. 10 сек)
- **jitter_sec** — случайная добавка к интервалу для вариативности
- **queue_order** — порядок обработки очередей (автоинкремент)

## 🗄 База данных

Приложение использует SQLite (`poster.db`) со следующими таблицами:

- **channels** — список каналов (id, chat_id, name)
- **queues** — очереди публикаций (параметры, статус, настройки)
- **post_log** — лог отправок (статус, ошибки, временные метки)

## 🆕 Новые функции

- **Умная отправка файлов** — автоматическое определение типа файла (фото, видео, GIF, документ) и выбор оптимального метода отправки
- **Jitter (случайная задержка)** — добавление случайной вариации к интервалу для естественности публикаций
- **Приоритет очередей** — настройка порядка обработки очередей, возможность изменения порядка «вверх/вниз»
- **Принудительный запуск** — немедленная активация очереди вне зависимости от расписания
- **ETA завершения** — расчёт предполагаемого времени окончания публикации очереди
- **Веб-интерфейс с автообновлением** — мониторинг статусов в реальном времени без перезагрузки страницы
- **Тестирование каналов** — быстрая проверка связи с каналом перед запуском очереди
- **Гибкое расписание** — настройка времени начала/окончания публикаций для каждого канала

## 🗓 В планах

### Ближайшие релизы

- [ ] **Асинхронная обработка очередей** — переход на индивидуальные задания APScheduler, устранение блокирующего `sleep`, каждая очередь работает независимо

- [ ] **Проверка размера файлов** — автоматический отсев файлов >10MB (фото) и >50MB (видео/документы) до отправки
- [ ] **Graceful shutdown** — обработка SIGTERM, корректная остановка планировщика, сохранение состояния БД
- [ ] **Защита от флуд-контроля Telegram** — повторные попытки с увеличением паузы при HTTP 429
- [ ] **Исправление тестового сообщения** — использование `getChat` вместо `sendMessage` для проверки канала
- [ ] **Миграции базы данных** — переход от DROP при каждом старте к системе миграций (Alembic) для сохранения данных при обновлениях

- [ ] **Валидация chat_id** — автоматическая проверка существования канала и прав бота при добавлении
- [ ] **Логирование в файл** — ротация логов, разделение по уровням (INFO, ERROR, DEBUG)

## ⚠️ Известные проблемы и ограничения

Проект находится в активной разработке. Ниже приведены известные проблемы, разделённые по приоритету.

### 🔴 Критические проблемы (требуют немедленного исправления)

| # | Проблема | Влияние | Статус |
|---|----------|---------|--------|
| 1 | **Блокирующий `time.sleep()` в process_queues** | Одна активная очередь блокирует все остальные каналы. При интервале 60 сек другие очереди ждут завершения сна | 🔄 В работе |
| 2 | **Гонки данных между auto_switch_queues и отправкой поста** | Два потока одновременно читают/пишут статусы очередей. Возможна отправка для очереди со статусом `ended` | 📋 Запланировано |
| 3 | **Бесконечная повторная отправка файла при ошибке** | При ошибке (отвал VPN, таймаут) файл не перемещается, хэш не помечается. После восстановления — 15–20 дублей | 📋 Запланировано |
| 4 | **Отсутствует проверка размера файла** | Лимиты Telegram: фото ≤10 MB, видео/документы ≤50 MB. Превышение вызывает ошибку и зацикливание | 📋 Запланировано |
| 5 | **Нет graceful shutdown** | При SIGTERM планировщик обрывается, БД может быть повреждена | 📋 Запланировано |
| 6 | **Риск повреждения SQLite при многопоточной работе** | Возможны ошибки `database is locked` при одновременной записи | 🔄 В работе |
| 7 | **Удаление очереди ломает ссылки `prev_queue_id`** | Паузынные очереди не восстанавливаются корректно | 📋 Запланировано |

### 🟠 Серьёзные недоработки (влияют на удобство и стабильность)

| # | Проблема | Влияние | Статус |
|---|----------|---------|--------|
| 8 | **Неэффективное автообновление страницы очередей** | Клиент шлёт `X-Requested-With`, но сервер возвращает полную страницу вместо фрагмента | 📋 Запланировано |
| 9 | **Нет защиты от флуд-контроля Telegram (429)** | При частых отправках можно получить временную блокировку | 📋 Запланировано |
| 10 | **Тестовое сообщение остаётся в канале** | `send_text("🔍 Проверка...")` видно подписчикам | 📋 Запланировано |
| 11 | **Некорректная фильтрация папки `sended`** | Папки с `sended` в имени ошибочно исключаются | 📋 Запланировано |
| 12 | **Отрицательный джиттер может дать паузу < 0** | Мгновенная отправка или ошибка планировщика | 📋 Запланировано |
| 13 | **Длительные транзакции SQLite из-за `sleep`** | Блокировка записи во время паузы | 🔄 В работе |
| 14 | **Нет валидации `chat_id` при добавлении канала** | Можно добавить несуществующий канал | 📋 Запланировано |
| 15 | **Хардкод путей к БД и медиа** | Невозможно запустить локально без правки кода | 📋 Запланировано |
| 16 | **Нет проверки существования папки `media`** | Тихие ошибки при отсутствии папки | 📋 Запланировано |
| 17 | **GIF отправляется как документ, а не анимация** | Файлы `.gif` не воспроизводятся как видео | 📋 Запланировано |

### 🟡 Ограничения разработки

| Проблема | Статус | Комментарий |
|----------|--------|-------------|
| Токен бота в репозитории | ✅ Осознанно | Только для разработки, 2 тестовых канала, будет удалено перед продакшеном |
| Нет аутентификации в веб-интерфейсе | ✅ Осознанно | Локальное использование, при сетевом доступе требуется фаервол |

## 📌 Как работает очередь постов

**Важно:** В текущей реализации все очереди обрабатываются в одном потоке с глобальным циклом `process_queues`. Это означает:

- ❌ **Пауза одной очереди блокирует все остальные** — если очередь A имеет интервал 60 сек, очереди B и C будут ждать завершения `sleep()`
- ❌ **Нет индивидуального планирования** — все очереди зависят от общего цикла

**План исправления (приоритет №1):**

1. Перейти на индивидуальные задания APScheduler для каждой активной очереди
2. Каждая очередь получит собственное задание с независимым интервалом
3. Удалить глобальный цикл `process_queues`
4. Пауза одной очереди **не будет влиять** на другие каналы

После исправления:
- ✅ Очередь канала А с интервалом 60 сек работает независимо
- ✅ Очередь канала Б с интервалом 30 сек работает параллельно
- ✅ Пауза/ошибка одной очереди не затрагивает остальные
